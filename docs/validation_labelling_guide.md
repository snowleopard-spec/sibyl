# Stage 3 — Layer 3 labelled validation guide

This is the hand-labelling workflow per spec §9.1.1. The goal is to
produce **ground-truth boundaries** for 20 hard-case filings, then
have Sibyl measure how close its extractor's boundaries are. Per
spec, this is "the only thing that proves trust" in Stage 3 output.

Time budget: ~30-60 min total (≈3 min per filing × 20).

---

## 1. The CSV already exists

`data/clean/validation_labels.csv` was generated by
`sibyl sections --pick-validation-set 20` and contains 20 filings
spanning hard cases:

- 4 filings with `yoy_jump` flags (likely parse breaks)
- 4 with `over_extracted` status (the Photronics failure mode)
- 4 oldest filings in the corpus (pre-2018 format risk)
- 4-ish random `ok` filings (baseline)
- Any duplicates between buckets are de-duped

Columns to fill in per row:

| Column | What to paste |
| --- | --- |
| `risk_factors_start_substring` | First ~30-50 chars where the **real** Risk Factors section begins |
| `risk_factors_end_substring` | First ~30-50 chars of whatever comes right after the section (e.g. "Item 1B.", "Item 2. Properties", "PART II") |
| `mdna_start_substring` | First ~30-50 chars where the **real** MD&A section begins |
| `mdna_end_substring` | First ~30-50 chars of whatever comes right after MD&A (e.g. "Item 7A.", "Item 8. Financial Statements") |
| `notes` | Anything weird about this filing (optional) |

If a filing has **no** Item 1A (older or shell-co filings) leave both
RF columns blank; same for MD&A. The validator handles blanks
gracefully.

---

## 2. How to label one filing

The substring approach is deliberate: it's robust to text length
differences and easy to copy/paste. You don't need character offsets.

For each row in the CSV:

1. Open the filing's `full.txt`:
   ```bash
   less data/clean/<CIK>/<ACCESSION>/full.txt
   ```
   In `less`, use `/` to search.

2. **Find the real Item 1A start.** Search for `Item 1A` and skip past
   any Table of Contents entries (usually clustered near the top).
   The real start typically reads "Item 1A. Risk Factors" followed by
   a real opening paragraph ("Investors should carefully consider...",
   "Our business is subject to..."). Copy the first ~30-50 chars
   *including* "Item 1A." into the `risk_factors_start_substring`
   column.

3. **Find Item 1A end.** Same idea — the next item header is the
   boundary. Search forward for `Item 1B` or `Item 2`. Copy ~30-50
   chars including the header.

4. **Repeat for Item 7 → 7A or Item 8.**

5. Save the CSV after every few rows so you don't lose work.

### Example labelled row

```csv
320193,0000320193-23-000106,2023-11-03,"Item 1A.    Risk Factors  The Company's business","Item 1B.    Unresolved Staff Comments","Item 7.    Management's Discussion and Analysis","Item 7A.    Quantitative and Qualitative Disclosures",
```

(Quotes around fields containing commas; spreadsheet apps handle this
automatically.)

---

## 3. Tips for the hard cases

- **Old filings (pre-2018):** Section headers often use ALL CAPS, no
  period: `ITEM 1A RISK FACTORS`. Search case-insensitively in `less`
  with `/-i Item 1A`.
- **"see Item 1A" or similar forward references:** these are NOT the
  section start. The real start has content after the header, not a
  cross-reference.
- **Incorporated by reference:** if the section says "Information
  required by this item is incorporated by reference from our 2024
  Proxy Statement," there is no real local section. Leave that
  section's start/end columns blank and note `incorp_ref` in `notes`.
- **The TOC trap:** the document's own Table of Contents will list
  "Item 1A. Risk Factors ... 27" at position ~2-5% of the doc. Skip
  past it.

## 4. Run the validator

Once the CSV is filled (you can validate partial labels — blank rows
are skipped):

```bash
.venv/bin/sibyl sections --validate
```

Output looks like:

```
== risk_factors ==
  labelled: 18
  recall:           18/18 = 100.00%
  start within 200: 17/18 = 94.44%
  end within 1000:  16/18 = 88.89%

== mdna ==
  labelled: 18
  recall:           18/18 = 100.00%
  start within 200: 16/18 = 88.89%
  end within 1000:  15/18 = 83.33%
```

### Gate criteria (must pass before Stage 4)

- **Section recall ≥ 90%** — extractor found *something* for each
  labelled section
- **Start-accuracy ≥ 85%** — detected start within 200 chars of the
  true start
- **End-accuracy** is informational only — section ends are fuzzier
  by nature

If either criterion fails:

1. Look at the failing rows. They're the only place the regex /
   library missed.
2. The fix is usually one of: bump `EXTRACTOR_VERSION` in
   `sibyl/sections.py` and tweak the failure-mode flagging; or wait
   for an `edgartools` upgrade.
3. `sibyl sections --force` re-runs the extractor on the whole
   corpus.

---

## 5. After the gate passes

Move on to Stage 4 (`sibyl score`) — L&M scoring per section using
the cleaned `risk_factors.txt` / `mdna.txt` files Sibyl has produced.

Filings with `parse_status='section_fail'` will still be scored via
`full.txt` only — they just won't contribute to per-section
signals. That's expected and built into the design (spec §8 Stage 3).

Keep the labelled CSV checked in (or backed up) so future re-runs
after an `EXTRACTOR_VERSION` bump can re-validate cheaply against
the same ground truth.
